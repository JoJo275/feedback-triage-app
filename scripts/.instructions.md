## Conventions (ADR 031)

### Structure

- Every script uses `argparse` for CLI with consistent flags: `--help`, `--version`, `--dry-run` (where applicable).
- Use `logging` for status/diagnostic messages (`logging.info()`, `logging.warning()`, `logging.error()`). Reserve `print()` only for primary output (tables, JSON, generated content).
- Configure logging in `main()` with `logging.basicConfig()`.
- Use `pathlib.Path` over `os.path` for all file operations.
- Use `subprocess.run()` with argument lists — never `shell=True`.
- Constants in `UPPER_SNAKE_CASE`.

### Shebang and Executable Bit

- Scripts with a shebang (`#!/usr/bin/env python3`) must be marked executable: `git add --chmod=+x scripts/my_script.py`.
- The `check-shebang-scripts-are-executable` pre-commit hook enforces this.

### Shared Modules

- Module names prefix with `_` to signal "internal library, not a CLI" — excluded from command reference generator.
- `_imports.py` — `import_sibling(name)` loads sibling scripts as modules. Use for cross-script imports:
    ```python
    from _imports import import_sibling
    _progress = import_sibling("_progress")
    ProgressBar = _progress.ProgressBar
    ```
- `_progress.py` — `ProgressBar` (determinate) and `Spinner` (indeterminate) with automatic Unicode/ASCII fallback. Import and use these instead of ad-hoc progress output.
    - **Colors**: Both `ProgressBar` and `Spinner` accept a `color=` argument (e.g. `"cyan"`, `"green"`). Set `DEFAULT_BAR_COLOR` or `DEFAULT_SPINNER_COLOR` module-level constants to apply a default without passing `color=` on every call.
    - **Pulse mode**: `ProgressBar(pulse=True)` keeps the bar continuously and smoothly advancing between `update()` calls by estimating step duration from observed timings. The fill creeps toward the next step boundary, recalculating speed as it learns how long each step takes. Use with context manager (`with ProgressBar(..., pulse=True) as bar:`).
    - **Keep-alive**: `Spinner(keep_alive=True)` is a documentation flag indicating the spinner's background thread always animates. The spinner already spins by default.
- `_colors.py` — `Colors` class + `supports_color()`, `colorize()`, `status_icon()` for consistent colored terminal output. Respects `NO_COLOR`/`FORCE_COLOR` env vars. Import and use these instead of ad-hoc ANSI escape codes.
- `_ui.py` — `UI` class for dashboard layout (headers, sections, footers, tables, key-value pairs). All CLI scripts should use `UI` for output instead of manual box-drawing characters.
    - **Recommended scripts**: Use `UI.recommended_scripts(keys, preamble=...)` instead of inline script lists. The `RECOMMENDED_SCRIPTS` registry in `_ui.py` is the single source of truth for script commands and descriptions. To add a new script to the registry, add an entry to the `RECOMMENDED_SCRIPTS` dict.
    - **Adding a recommended section**: Call `ui.recommended_scripts(["key1", "key2"], preamble="...")` at the end of your script's main output. Pick keys relevant to the script's domain.

### Common Conventions for Script Output

- **Progress indicators**: Always use `ProgressBar` or `Spinner` from `_progress.py` for long-running operations. Use `pulse=True` on `ProgressBar` when a step may block for a while without calling `update()`.
- **Recommended scripts section**: Add a recommended scripts section at the end of scripts that produce diagnostic or informational output. Use the shared `UI.recommended_scripts()` method. Descriptions appear as colored mini-titles above commands.
- **Markdown config workflow**: Scripts with complex configuration (like `customize.py` and `git_doctor.py`) support `--export-config` / `--apply-from` for offline editing via Markdown files. Follow the same pattern: export generates a Markdown document with editable fields, user edits it, `--apply-from` parses and applies.
- **Logging**: Use `logging.info()` for status messages, `logging.warning()` for warnings, `logging.error()` for errors. Configure in `main()` with `logging.basicConfig(format="%(message)s", level=level)`. Reserve `print()` for primary output.
- **Dashboard theme**: Each script sets `THEME = "color"` for consistent accent coloring via `UI(theme=THEME)`.
- **Version constant**: Each script defines `SCRIPT_VERSION = "x.y.z"` for `--version` flag and UI header.

### Dead Code Cleanup

When refactoring scripts (e.g., migrating inline UI logic to `_ui.py`,
extracting helpers to shared modules), always verify that old inline
implementations are fully removed from the migrated scripts. Common
leftovers after shared-module migrations:

- Padding/alignment helpers replaced by `UI.table_row()`
- Color-wrapping functions replaced by `Colors` methods
- Box-drawing character setup replaced by `UI` constructor
- Progress-indicator logic replaced by `ProgressBar`/`Spinner`

After any refactor, grep for the old function name across `scripts/`
to confirm no callers remain before considering the migration complete.

## Ruff and Bandit Compliance

Scripts run through **ruff** (lint + format) and **bandit** on every commit
via pre-commit hooks. Write code that passes both on the first try.

### Key ruff rules for scripts

- **PERF401**: Use `list.extend(gen)` or a comprehension — not a
  for-loop that appends one item at a time.
- **C4**: Prefer list/dict/set comprehensions over `list(generator)`.
- **SIM**: Simplify boolean logic, use ternary where clear.
- **B**: No mutable default args, no bare `except`.
- **PTH**: Use `pathlib.Path` everywhere — `Path.open()` not `open()`.
- **UP**: Modern Python 3.11+ syntax (`X | Y`, f-strings, `dict` not `Dict`).
- **T20** and **D** are disabled for scripts via per-file-ignores, so
  `print()` and missing docstrings are fine.

### Bandit rules for scripts

- **B602/B603**: Never use `subprocess.run(shell=True)`. Pass arg lists.
- **B307**: Never use `eval()` or `exec()`.
- **B108**: Don't hardcode `/tmp`. Use `tempfile`.
- **B506**: Use `yaml.safe_load()`, never bare `yaml.load()`.
- **B101**: Suppressed globally — asserts are allowed.

### Formatting

- ruff format enforces double quotes, trailing commas in multi-line
  structures, and 88-char target line length.
- Don't fight the formatter — write code in its style.

## Python Style

- Absolute imports: `from feedback_triage.module import func`
- Type hints on all public functions and methods.
- Docstrings in Google style format.
- `tomllib` (stdlib 3.11+) for TOML reading — no third-party TOML lib.
- `importlib.metadata` for installed-package introspection.
- `shutil.get_terminal_size()` for terminal dimensions.

## Required Script Conventions

The rules below are enforced for every new script and for any existing
script that gets non-trivial edits. Rationale lives in
[`docs/notes/script-conventions.md`](../docs/notes/script-conventions.md).

### 1. Standard module skeleton

Every CLI script opens in this order so navigation is muscle memory:

```python
#!/usr/bin/env python3
"""One-line summary.

Longer description.

Usage::
    python scripts/foo.py [--flags]
"""

from __future__ import annotations

# 1. stdlib
import argparse
import logging
import sys

# 2. third-party

# 3. project
from feedback_triage.x import y

# 4. local script modules (underscore-prefixed)
from _imports import find_repo_root, import_sibling
from _ui import UI, ExitCode

logger = logging.getLogger(__name__)

# --- Constants ---
SCRIPT_VERSION = "1.0.0"
THEME = "cyan"
ROOT = find_repo_root()


# --- Helpers ---
def _do_thing() -> None: ...


# --- CLI ---
def main(argv: list[str] | None = None) -> int: ...


if __name__ == "__main__":
    sys.exit(main())
```

### 2. `--format {text,json}` instead of ad-hoc `--json`

One flag, two values. JSON output skips `UI` chrome and emits a single
object on stdout for piping. Scripts that previously took `--json`
should accept both for one release, then drop `--json`.

### 3. Standard exit codes via `ExitCode`

Return values from `main()` use the [`ExitCode`](../scripts/_ui.py)
enum:

| Member     | Value | Meaning                                 |
| ---------- | ----- | --------------------------------------- |
| `OK`       | 0     | Success.                                |
| `WARN`     | 1     | Ran successfully but surfaced warnings. |
| `FAIL`     | 2     | Logical failure.                        |
| `USAGE`    | 64    | Bad CLI arguments.                      |
| `INTERNAL` | 70    | Unexpected internal error.              |

CI gates can then distinguish "lint warned" from "lint exploded."

### 4. `--quiet` / `--verbose` are universal

Two flags, three levels. Wire them to logging in `main()`:

```python
parser.add_argument("-q", "--quiet", action="count", default=0)
parser.add_argument("-v", "--verbose", action="count", default=0)
level = logging.WARNING - 10 * args.verbose + 10 * args.quiet
logging.basicConfig(level=level, format="%(message)s")
```

### 5. `--smoke` self-check contract

Every CLI script implements `--smoke`. The smoke run must:

- Make **no** network or DB calls.
- Validate all module-level constants and data structures.
- Exit `ExitCode.OK` on success, print `<name> <SCRIPT_VERSION>: smoke ok`.

`scripts/smoke_all.py` (when added) walks `scripts/*.py` and runs each
with `--smoke` as a single CI step.

### 6. Side-effect scripts declare `--dry-run`

If a script writes anywhere (DB, filesystem outside its own output,
remote API), it accepts `--dry-run` and the dry-run path is the
**default** in unsure cases. `seed.py` and `apply_labels.py` are the
reference implementations.

### 7. Long-running steps use `pulse=True`

Steps that may block for >2s without calling `bar.update()` use
`ProgressBar(..., pulse=True)` so the bar still animates. Avoids the
"is it hung?" problem in CI logs and slow VMs.

### 8. Env-var reads live in one helper per script

No `os.environ.get("FOO", "")` scattered through a script. Read
required env vars once, near the top of the module, with explicit
defaults and types:

```python
def _load_env() -> tuple[str, int]:
    db_url = os.environ.get("DATABASE_URL", "")
    timeout = int(os.environ.get("TIMEOUT", "30"))
    return db_url, timeout
```

Easier to mock in `--smoke`; one place to document defaults.

### 9. One CLI surface per script — no hidden subcommands

If a script grows a second mode that needs more than one or two flags,
**split it into a second script** rather than adding `argparse`
subparsers. Subcommands hide capabilities behind `--help` digging and
complicate the recommended-scripts registry. Two scripts named for what
they do beat one script with `foo serve` / `foo migrate` / `foo seed`.

### 10. Idempotency by default; destructiveness behind a flag

Running a script twice leaves the system in the same state as running
it once. Anything that destroys data (truncate, force-push, delete
file) requires an explicit flag (`--reset`, `--force`) **and** refuses
to act if the destructive flag is the only signal of intent. `seed.py`'s
"refuse if non-empty unless `--force` or `--reset`" is the pattern.

### 11. `task` shortcut for every user-facing script

If a human is expected to run it more than once, it gets a `Taskfile.yml`
entry. Internal scripts (called by other scripts or by CI only) stay
out of `Taskfile.yml` to keep `task --list` short. Litmus test: "would
a new contributor look for this in `task --list`?"

### 12. Shared modules stay shared — don't fork

When two scripts grow the same helper, extract to `_<name>.py` instead
of copy-pasting. The `_`-prefix convention already excludes them from
the command-reference generator. Helpers stay small and single-purpose;
when one starts hosting a class hierarchy, that's a sign it should move
into `src/feedback_triage/` or its own package.

### 13. Output is line-oriented and grep-friendly

Even with `UI` chrome, the load-bearing lines (`PASS`, `FAIL`, the
JSON blob, file paths) are on their own lines, prefixable, and contain
no Unicode characters that ASCII grep won't match. Pretty boxes are
_around_ the data, not woven into it.

### 14. Errors include the next action

When a script exits non-zero, the last line tells the caller what to
do — not just what's wrong. _"DATABASE_URL not set; export it or copy
`.env.example` to `.env`"_ beats _"DATABASE_URL not set."_

### 15. Every script's `--help` shows real examples

Use `argparse`'s `epilog=`. Two or three concrete invocations at the
bottom of `--help` save more time than a paragraph of prose:

```python
parser = argparse.ArgumentParser(
    description="Seed feedback_item with demo rows.",
    epilog=(
        "Examples:\n"
        "  python scripts/seed.py --dry-run\n"
        "  python scripts/seed.py --reset\n"
        "  python scripts/seed.py --force   # adds duplicates\n"
    ),
    formatter_class=argparse.RawDescriptionHelpFormatter,
)
```

### 16. Color is decoration, never the only signal

`Colors` and `status_icon()` already respect `NO_COLOR`. The _meaning_
is in the text too — never rely on red-vs-green alone. `status_icon()`
gives the icon; the icon plus the word (`FAIL: …`) is the rule.

### 17. Name boolean flags for the action, not the state

`--no-color` (action) is clearer than `--monochrome` (state).
`--force` (action) beats `--unsafe` (state). Verbs in the flag name
let `--help` read as a list of things the script can do.
